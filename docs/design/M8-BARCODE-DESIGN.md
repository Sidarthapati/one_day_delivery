## 1. Purpose

M8 owns the physical-digital identity of every parcel on the platform. Specifically:

- **Parcel ID generation** — globally unique, structured, human-readable
- **Label generation** — printable QR codes for both individual parcels and bags
- **Scan event ledger** — append-only, tamper-evident log of every physical scan at every point in the journey
- **Barcode lookup API** — given a parcel ID, return current state and full scan history

M8 is a utility module: it has no opinion about logistics decisions. It receives instructions from other modules (M4 assigns an ID, M7 assigns a stand) and records what physically happened (scans). It never initiates a workflow — it only records and serves data.

---

## 2. Scope

### In scope for v1
- Structured parcel ID generation (city + date + sequence)
- QR code label generation (parcel label + bag label)
- Append-only scan event ledger
- Barcode lookup API (current state + history)
- Scan ingestion from DA app, hub scan guns, airport handhelds
- Online-only scanning (no offline queue in v1)

### Out of scope for v1
- Offline scan queue with reconnect sync (deferred — see §12)
- Label template customisation per B2B account (later phase)
- Scan event idempotency strategy (open decision — see §12)
- Physical label printing integration (M8 produces the data; printer integration is a client concern)

## 3. Parcel ID Format

### 3.1 Structure

```
{CITY_CODE}-{YYYYMMDD}-{6-digit-sequence}
```

**Examples:**
- `BLR-20260510-000042` — 42nd parcel booked in Bengaluru on 10 May 2026
- `BOM-20260510-001337` — 1337th parcel booked in Mumbai on 10 May 2026

**City codes (v1 cities):**

| City | Code |
|---|---|
| Bengaluru | `BLR` |
| Mumbai | `BOM` |
| Delhi | `DEL` |
| Hyderabad | `HYD` |
| Chennai | `MAA` |

### 3.2 Generation Rules

- Sequence resets to `000001` at **midnight IST** per city per day
- Sequence is zero-padded to 6 digits (max 999,999 parcels per city per day; widen to 7 if needed)
- Date component uses the **booking date in IST**, not UTC, so a parcel booked at 11:30 PM IST gets that day's date even though UTC is next day
- The full parcel ID is **immutable** once assigned — it never changes for the lifetime of the parcel

### 3.3 ID Generation Implementation

IDs are generated by M4 at booking time and passed to M8. M8 does not generate IDs independently — it only receives and indexes them.

```sql
-- Lives in M4 (not M8)
CREATE TABLE parcel_id_counters (
  city_code  VARCHAR(10) NOT NULL,
  date_key   DATE NOT NULL,       -- date in IST (handled at application layer)
  next_val   INTEGER NOT NULL DEFAULT 1,
  PRIMARY KEY (city_code, date_key)
);
```

**Atomic increment (application layer):**

```java
// SELECT next_val FROM parcel_id_counters WHERE city_code=? AND date_key=? FOR UPDATE
// UPDATE parcel_id_counters SET next_val = next_val + 1 WHERE ...
// Return: String.format("%s-%s-%06d", cityCode, dateKey, acquiredVal)
```

## 4. Label Design

### 4.1 Parcel Label (QR Code)

The parcel label is attached by the DA at first-mile pickup. It is the single physical identity of the parcel for its entire journey.

**QR Code payload (JSON, encoded as QR):**

```json
{
  "v": 1,
  "pid": "BLR-20260510-000042",
  "dest_hub": "BOM-HUB-01",
  "dest_city": "BOM",
  "dest_pincode": "400002",
  "sort_key": "BOM-400002",
  "shipment_ref": "1DD-BLR-20260510-00042"
}
```

**Field notes:**
- `v` — payload version; allows format evolution without breaking old scanners
- `pid` — the parcel ID; the primary lookup key
- `dest_hub` — destination hub ID; used at origin hub to route to correct stand
- `sort_key` — city + pincode composite; used at destination hub for last-mile sortation
- `shipment_ref` — human-readable reference for manual lookup

**QR Code specification:**
- Error correction level: **M** (15% error recovery — balances data density and damage tolerance)
- Minimum module size: **3×3 pixels per cell** at 300 DPI for thermal printer compatibility
- Quiet zone: 4 modules on all sides
- Format: PNG (for label rendering); SVG on request for vector output

**Human-readable zone below the QR (printed text):**
```
BLR-20260510-000042
BOM | 400002
1DD-BLR-20260510-00042
```

### 4.2 Bag Label (QR Code)

Bag labels are generated by M7 after bag creation. M8 provides the QR generation utility; M7 supplies the data.

**Bag QR Code payload:**

```json
{
  "v": 1,
  "bag_id": "BAG-BLR-20260510-00007",
  "flight_no": "6E-101",
  "flight_date": "2026-05-10",
  "origin_hub": "BLR-HUB-01",
  "dest_hub": "BOM-HUB-01",
  "stand_no": "A-12",
  "parcel_count": 34,
  "parcel_ids": [
    "BLR-20260510-000001",
    "BLR-20260510-000002"
    // ... up to parcel_count entries
  ]
}
```

**Note:** `parcel_ids` array is embedded in the QR for small bags. For bags with > 100 parcels, the QR payload omits the array and carries only a `manifest_url` reference instead (to stay within QR data limits).

## 5. Scan Event Ledger

### 5.1 Design Principles

- **Append-only:** scan records are never updated or deleted
- **Source of truth:** the ledger is the physical audit trail; it is the ground truth for SLA computation (M10) and customer tracking (M4)
- **Immutable timestamps:** `occurred_at` is set by the scanning device; `received_at` is set by the server on ingestion; both are stored and neither is altered

### 5.2 Scan Points

| Scan Point | Code | Actor | Notes |
|---|---|---|---|
| DA label attach (pickup) | `DA_PICKUP_ATTACH` | DA app | First scan; establishes parcel identity |
| DA en route to hub | `DA_HUB_HANDOFF` | DA app | DA hands parcel to hub/van |
| Origin hub in-scan | `HUB_ORIGIN_IN` | Hub scan gun | Triggers M4 → AT_ORIGIN_HUB |
| Bag seal | `HUB_BAG_SEALED` | Hub scan gun | Parcel locked into bag |
| Airport handover | `AIRPORT_HANDOVER` | Airport handheld | Bag handed to GHA |
| GHA acceptance | `GHA_ACCEPTANCE` | GHA / airline system | Confirms airline took custody |
| Flight departed | `FLIGHT_DEPARTED` | M9 (system scan) | System-generated; not physical |
| Destination hub in-scan | `HUB_DEST_IN` | Hub scan gun | Triggers M4 → AT_DEST_HUB |
| Last-mile DA pickup | `DA_LASTMILE_PICKUP` | DA app | DA takes parcel for delivery |
| Delivery confirmation | `DA_DELIVERY_CONFIRM` | DA app | Triggers M4 → DELIVERED |
| Delivery failed | `DA_DELIVERY_FAILED` | DA app | Triggers M4 → DELIVERY_FAILED |

### 5.3 Scan Event Entity

```
ScanEvent
  id              UUID        PK
  parcel_id       VARCHAR(30) indexed; FK-like reference (not a hard FK for append-only safety)
  bag_id          VARCHAR(30) nullable; set for bag-level scans
  scan_point      VARCHAR(50) one of the codes above
  actor_id        UUID        who performed the scan (DA, hub operator, system)
  actor_role      VARCHAR(50) role at time of scan; denormalised for audit
  city_id         VARCHAR(10) city where scan occurred
  hub_id          VARCHAR(50) nullable; hub identifier if at a hub
  device_id       VARCHAR(100) nullable; scan gun or device identifier
  gps_lat         DECIMAL(9,6) nullable; GPS at time of scan (DA app)
  gps_lng         DECIMAL(9,6) nullable
  occurred_at     TIMESTAMPTZ when the physical scan happened (device clock)
  received_at     TIMESTAMPTZ when the server received and persisted it (server clock)
  raw_payload     JSONB       nullable; full raw event for debugging
```

### 5.4 Idempotency

> **Open Decision OD-M8-1:** Scan event idempotency strategy is not yet decided.
>
> **Options to evaluate before implementation:**
>
> **Option A — Dedup at write by (parcel_id + scan_point + occurred_at):**  
> Duplicate submissions are silently dropped. Clean ledger. Risk: clock skew on devices can cause legitimate scans to be dropped.
>
> **Option B — Accept all, mark duplicates:**  
> All scans recorded. A `is_duplicate` boolean flags suspicious duplicates (same parcel + scan_point within a configurable window, e.g. 30 seconds). Useful for debugging hardware issues.  
>
> **Option C — Client-supplied idempotency key:**  
> Scanning device generates a UUID for each scan event. Server deduplicates on that key. Clean ledger, no clock dependency.  
>
> **Recommendation:** Option C is the most robust. Each scan event submission includes a client-generated `idempotency_key` (UUID). The server stores it with a UNIQUE constraint and returns `200 OK` (not `409`) on duplicate submission (idempotent by definition). Add this to the schema now even if not enforced immediately.

```sql
-- Additional column on scan_events:
idempotency_key  VARCHAR(100)  UNIQUE
```

## 6. API Design

### 6.1 Scan Ingestion

#### `POST /api/v1/scans` — Submit a scan event

**Auth:** JWT (DA app, hub operator, airport device)

**Request:**
```json
{
  "idempotency_key": "device-uuid-client-generated",
  "parcel_id": "BLR-20260510-000042",
  "bag_id": null,
  "scan_point": "HUB_ORIGIN_IN",
  "occurred_at": "2026-05-10T13:04:55+05:30",
  "hub_id": "BLR-HUB-01",
  "gps_lat": 12.9716,
  "gps_lng": 77.5946
}
```

**Flow:**
1. Validate JWT and extract `actor_id`, `actor_role`, `city_id`
2. Validate `parcel_id` exists in ledger (i.e. has been previously seen via `DA_PICKUP_ATTACH`)
3. Validate `scan_point` is a known code
4. Check idempotency key; return `200` if already processed
5. Persist `ScanEvent`
6. Publish `scan.event` Kafka message
7. Return `201 Created`

**Response `201 Created`:**
```json
{
  "scan_id": "uuid",
  "parcel_id": "BLR-20260510-000042",
  "scan_point": "HUB_ORIGIN_IN",
  "received_at": "2026-05-10T07:35:02Z"
}
```

**Error responses:**
| Code | Scenario |
|---|---|
| `404` | `parcel_id` has never been registered (no `DA_PICKUP_ATTACH` scan) |
| `400` | Unknown `scan_point` code |
| `401` | Invalid or expired JWT |
| `403` | Actor role not allowed to submit this scan point |

---

#### `POST /api/v1/scans/batch` — Bulk scan submission (hub scan guns)

Hub operators may scan many parcels in rapid succession. Batch endpoint accepts up to **100 scan events** per request.

**Request:**
```json
{
  "scans": [
    { "idempotency_key": "...", "parcel_id": "BLR-20260510-000042", "scan_point": "HUB_ORIGIN_IN", "occurred_at": "..." },
    { "idempotency_key": "...", "parcel_id": "BLR-20260510-000043", "scan_point": "HUB_ORIGIN_IN", "occurred_at": "..." }
  ]
}
```

**Response:** Per-item result array. Partial success is allowed — each item has its own `status` and `error` field.

### 6.2 Label Generation

#### `GET /api/v1/labels/parcel/{parcel_id}` — Get parcel QR label

**Auth:** Internal service token or DA app JWT

**Query params:**
- `format=png` (default) | `svg` | `json` (raw payload only)
- `size=200` — QR image size in pixels (default 200)

**Response:** Binary PNG or SVG, or JSON payload depending on `format`.

---

#### `GET /api/v1/labels/bag/{bag_id}` — Get bag QR label

**Auth:** Hub operator JWT or internal service token

Same format params as parcel label.

### 6.3 Lookup

#### `GET /api/v1/barcodes/{parcel_id}` — Lookup parcel state and history

**Auth:** None (public for customer use) or JWT for internal detail level

**Response `200 OK`:**
```json
{
  "parcel_id": "BLR-20260510-000042",
  "shipment_ref": "1DD-BLR-20260510-00042",
  "current_scan_point": "HUB_ORIGIN_IN",
  "last_scanned_at": "2026-05-10T13:04:55+05:30",
  "scan_history": [
    {
      "scan_point": "DA_PICKUP_ATTACH",
      "scan_point_label": "Label attached at pickup",
      "occurred_at": "2026-05-10T11:20:00+05:30",
      "city": "BLR"
    },
    {
      "scan_point": "HUB_ORIGIN_IN",
      "scan_point_label": "Arrived at Bengaluru hub",
      "occurred_at": "2026-05-10T13:04:55+05:30",
      "city": "BLR",
      "hub_id": "BLR-HUB-01"
    }
  ]
}
```

Public callers see only `scan_point_label` (human-readable) and timestamps. Internal callers with JWT see full fields including `actor_id`, `device_id`, and `gps_lat/lng`.

## 7. Kafka Event Schema

### 7.1 Events Produced by M8

#### `scan.event`
**Topic:** `oneday.scan.events`  
**Key:** `parcel_id`

```json
{
  "event_id": "uuid",
  "occurred_at": "2026-05-10T07:34:55Z",
  "received_at": "2026-05-10T07:35:02Z",
  "parcel_id": "BLR-20260510-000042",
  "bag_id": null,
  "scan_point": "HUB_ORIGIN_IN",
  "actor_id": "actor-uuid",
  "actor_role": "HUB_OPERATOR",
  "city_id": "BLR",
  "hub_id": "BLR-HUB-01"
}
```

**Consumers:**
- M4 — state machine transition (e.g. `HUB_ORIGIN_IN` → AT_ORIGIN_HUB)
- M7 — hub workflow triggers
- M10 — SLA leg timestamp

**Topic partitioning:** Partition by `parcel_id` hash. Guarantees scan events for one parcel are processed in order by consumers.

## 8. Database Schema

### 8.1 Flyway Migration: `V1__create_scan_events.sql`

```sql
CREATE TABLE scan_events (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  idempotency_key  VARCHAR(100) UNIQUE,
  parcel_id        VARCHAR(30) NOT NULL,
  bag_id           VARCHAR(30),
  scan_point       VARCHAR(50) NOT NULL,
  actor_id         UUID NOT NULL,
  actor_role       VARCHAR(50) NOT NULL,
  city_id          VARCHAR(10) NOT NULL,
  hub_id           VARCHAR(50),
  device_id        VARCHAR(100),
  gps_lat          DECIMAL(9, 6),
  gps_lng          DECIMAL(9, 6),
  occurred_at      TIMESTAMPTZ NOT NULL,
  received_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  raw_payload      JSONB,
  is_duplicate     BOOLEAN NOT NULL DEFAULT FALSE
);

-- Primary lookup: all scans for a parcel in order
CREATE INDEX idx_scan_events_parcel_id    ON scan_events(parcel_id, occurred_at);

-- Hub operations: all scans at a hub in a time window
CREATE INDEX idx_scan_events_hub_time     ON scan_events(hub_id, occurred_at) WHERE hub_id IS NOT NULL;

-- SLA queries: all scans for a city in a window
CREATE INDEX idx_scan_events_city_time    ON scan_events(city_id, occurred_at);

-- Actor audit queries
CREATE INDEX idx_scan_events_actor        ON scan_events(actor_id, occurred_at);

-- Bag-level queries
CREATE INDEX idx_scan_events_bag_id       ON scan_events(bag_id) WHERE bag_id IS NOT NULL;
```

**Retention note:** `scan_events` is the primary audit log. Rows are **never deleted**. Partition by month after the table exceeds 50M rows (PostgreSQL declarative partitioning on `occurred_at`).

## 10. QR Code Generation

**Library:** `com.google.zxing:core` + `com.google.zxing:javase` (ZXing)

**QR generation utility (internal, package-private):**

```java
// Pseudocode
byte[] generateQr(String payload, int sizePx, ErrorCorrectionLevel ecLevel) {
    QRCodeWriter writer = new QRCodeWriter();
    BitMatrix matrix = writer.encode(payload, BarcodeFormat.QR_CODE, sizePx, sizePx,
        Map.of(EncodeHintType.ERROR_CORRECTION, ecLevel,
               EncodeHintType.MARGIN, 4));
    BufferedImage image = MatrixToImageWriter.toBufferedImage(matrix);
    // convert to PNG bytes
}
```

**Caching:** QR label bytes are **cached in Redis** with key `qr:parcel:{parcel_id}` and TTL of 24 hours. Labels rarely change; caching avoids recomputation on repeat fetches by DA apps.

## 11. Integration Points

| Module | How M8 integrates |
|---|---|
| **M4 (Orders)** | M4 calls `BarcodeService.registerParcel()` after booking; M8 emits `scan.event` consumed by M4 for state transitions |
| **M5 (DA Dispatch)** | DA app submits scans via M8 REST API; `DA_PICKUP_ATTACH`, `DA_DELIVERY_CONFIRM`, `DA_DELIVERY_FAILED` |
| **M7 (Hub Ops)** | Hub guns submit scans via M8 batch API; M7 calls `BarcodeService.buildBagLabel()` |
| **M9 (Airline)** | Airport handhelds submit `AIRPORT_HANDOVER` and `GHA_ACCEPTANCE` via M8; M9 generates system-level `FLIGHT_DEPARTED` scan |
| **M10 (SLA)** | Consumes `oneday.scan.events` Kafka topic; calls `BarcodeService.getLatestScanAtPoint()` for SLA queries |

## 12. Open Decisions

| ID | Decision | Impact | Recommended path |
|---|---|---|---|
| OD-M8-1 | Scan idempotency strategy | Ledger correctness | Implement Option C (client idempotency key) from day one; cheapest to add now, expensive to retrofit |
| OD-M8-2 | Offline scan queue | DA app reliability in low-signal areas | Online-only in v1; design DA app with a queued retry layer so scans are buffered on device and replayed when online |
| OD-M8-3 | QR vs Code128 for bag labels | Scanner hardware compatibility | QR chosen; validate with hub scan gun hardware before printing at scale |
| OD-M8-4 | Barcode standard with airline partners | Airport scanner compatibility | Check IATA standard before finalising bag label format; may need DataMatrix for airline scanners |
| OD-M8-5 | Scan event table partitioning strategy | Long-term DB performance | Partition by `occurred_at` month; implement from day one on PostgreSQL 14+ (zero migration cost later) |
| OD-M8-6 | Label template per B2B account | Customer branding | Out of scope v1; design `LabelBuilder` as a strategy pattern so templates are pluggable |
