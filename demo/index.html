Safe Retries / Idempotency

    
Every request carries an Idempotency-Key. If the same key is sent again — due to a network retry, double-click, or app crash — the backend returns the exact same response instead of creating a duplicate shipment.

Flow

      

        
1Booking created → state: BOOKED

        
2M5 assigns a DA → state: PICKUP_ASSIGNED + OTP generated & SMS'd

        
3DA arrives, asks sender for OTP, enters it in the DA app

        
4API verifies OTP → state: PICKED_UP

      

    


    

      
      
    


    

    

      
      
    

    
  

  

    

      
Resend Endpoint

      

        POST
        /internal/v1/shipments/{ref}/pickup-otp/resend
      

      
200 OK  →  {"otp": "4821"}  — new OTP (also sent via SMS)
404     →  Shipment not found
409     →  Not in PICKUP_ASSIGNED state
429     →  Resend limit (3) reached

    


    

      
Verify Endpoint

      

        POST
        /internal/v1/shipments/{ref}/pickup-otp/verify
      

      
Body: {"otp": "4821"}

      
204 No Content → OTP correct; PICKUP_ASSIGNED → PICKED_UP
404     →  Shipment not found
409     →  Not in PICKUP_ASSIGNED state
422     →  OTP wrong / expired / already used
